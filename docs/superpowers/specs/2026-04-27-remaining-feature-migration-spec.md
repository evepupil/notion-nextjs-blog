# Remaining Feature Migration Spec

> Status: Draft ready for implementation planning
> Date: 2026-04-27
> Source project: `D:\myproject\my-fuwari`
> Target project: `D:\myproject\notion-nextjs-blog`

## Goal

Migrate the useful capabilities from the old Fuwari/Astro blog into the new Next.js platform without copying the old theme architecture or visual style.

The new platform should preserve content value, SEO value, and publishing workflow, while making room for books, tools, lab experiments, friends, sponsor/about pages, search, and future AI features.

## Already Migrated

- Markdown post source under `content/posts/{zh,en}`.
- Notion sync script adapted for localized posts and image paths.
- Localized article routes: `/zh/posts/[slug]/`, `/en/posts/[slug]/`.
- Legacy article redirects through Cloudflare Pages `_redirects`:
  - `/posts/[slug]/` -> `/zh/posts/[slug]/`
  - `/posts/en/[slug]/` -> `/en/posts/[slug]/`
- Basic sitemap and robots generation.
- Static Cloudflare Pages deployment through `out`.
- Basic Markdown rendering, heading extraction, JSON-LD, canonical, hreflang metadata.

## Remaining Old-Site Capabilities

From old project files and routes:

- Archive and tag pages: `src/pages/archive/**`.
- About pages: `src/pages/about.astro`, `src/pages/about/en.astro`.
- Friends pages and friend submission API: `src/pages/friends.astro`, `src/pages/friends/en.astro`, `src/pages/api/_submit-friend-link.ts`.
- Sponsors pages: `src/pages/sponsors.astro`, `src/pages/sponsors/en.astro`.
- Books pages and chapter routes: `src/pages/books/index.astro`, `src/pages/books/[book]/[chapter].astro`.
- Search data endpoint: `src/pages/api/search-data.json.ts`.
- RSS feed: `src/pages/rss.xml.ts`.
- Post widgets: TOC, license, language switcher, navbar panel, back-to-top, pagination.
- Analytics config: Umami public share link and script config.
- Image fallback config from `eo-r2.2x.nz` to R2 public domain.
- GitHub edit/source link config.
- Code block enhancements from Expressive Code plugins.

## Migration Principles

- Do not recreate Fuwari's layout or theme system.
- Keep routes localized under `/zh` and `/en` for new canonical pages.
- Preserve only article legacy redirects for now unless a public external link requires another legacy path.
- Prefer static generation for SEO-critical pages.
- Edge/API features must be optional and degrade gracefully on static deployment.
- Each feature should own its data model under `features/<feature>` and keep page files thin.

## Feature Specs

### 1. Archive and Tags

Target routes:

- `/zh/archive/`
- `/en/archive/`
- `/zh/tags/[tag]/`
- `/en/tags/[tag]/`

Requirements:

- Archive groups posts by year and month.
- Tags page lists all tags with post counts.
- Individual tag page lists matching posts using the redesigned post card.
- Pinned posts should not break chronological archive ordering; archive sorts strictly by date desc.
- Metadata must include canonical and localized alternates where possible.

Data helpers:

- `features/posts/lib/archive.ts`
- `features/posts/lib/tags.ts`

Acceptance:

- Generated static pages for all tags in Markdown frontmatter.
- No runtime `fs` access after static export.

### 2. RSS and Search Index

Target files/routes:

- `/rss.xml`
- `/search-data.json`

Requirements:

- RSS includes title, description, canonical URL, publication date, and sanitized HTML or description fallback.
- RSS image URLs should resolve to absolute site URLs when possible.
- Search index contains title, description, slug, locale, URL, tags, published date, category, and excerpt.
- Search index must be static and generated during build.

Implementation options:

- Prefer Next metadata route or static generation script if `output: "export"` has limitations.
- If route handlers complicate static export, generate files into `out` from a post-build script.

Acceptance:

- `out/rss.xml` exists after `bun run pages:build`.
- `out/search-data.json` exists after `bun run pages:build`.
- Files do not expose drafts.

### 3. About and Profile

Target routes:

- `/zh/about/`
- `/en/about/`

Requirements:

- Profile information from old config: author name `叶桐`, bio, GitHub, Zhihu, avatar candidates.
- Page should be personal but concise: identity, current focus, links, site philosophy.
- Avatar handling should use stable local/public assets if available; otherwise use monogram fallback.
- Include SEO metadata and localized copy.

Acceptance:

- Header nav can include About later, but v1 may expose it from footer/home.
- No dependency on old Astro config file.

### 4. Friends

Target routes:

- `/zh/friends/`
- `/en/friends/`

Data:

- Prefer `data/friends.json` generated by Notion sync.
- Keep old `sync-friends` script behavior but adapt fields to the new feature model.

Requirements:

- Friends list displays name, URL, avatar, description, status/category if available.
- Submission form is postponed until Edge/Worker strategy is stable.
- Static page should explain how to request a friend link, using GitHub issue/email/manual instructions.

Acceptance:

- Static friends page builds without secrets.
- Missing data shows an intentional empty state.

### 5. Books

Target routes:

- `/zh/books/`
- `/en/books/`
- `/zh/books/[book]/`
- `/en/books/[book]/`
- `/zh/books/[book]/[chapter]/`
- `/en/books/[book]/[chapter]/`

Requirements:

- Books support title, description, cover, tags, status, chapter count, updated date.
- Chapters render Markdown with the same reader system as posts, but with book navigation.
- Book content should live under `content/books/{zh,en}` or a clearly documented equivalent.
- Do not block frontend redesign on full book migration; first implement listing placeholders with real data model.

Acceptance:

- Books list no longer shows placeholder copy.
- At least one fixture or migrated book can render in static export before enabling full sync.

### 6. Tools and Lab

Target routes:

- `/zh/tools/`, `/en/tools/`
- `/zh/lab/`, `/en/lab/`

Requirements:

- Tools are stable utilities or links the visitor can use.
- Lab is for experiments, prototypes, Workers, AI demos, and build logs.
- Both use data-driven registries rather than hardcoded scattered JSX.

Data models:

- `features/tools/types.ts`
- `features/lab/types.ts`
- `data/tools.json`
- `data/lab.json`

Acceptance:

- Pages list entries with status (`可用`, `建设中`, `归档`) and links.
- Empty state looks intentional.

### 7. Ask AI

Target route:

- `/zh/ask/`, `/en/ask/`

Requirements:

- v1 static page explains planned AI Search capability and links to posts/search.
- Full AI Q&A should wait until deployment strategy for Pages Functions/Workers is decided.
- Future dynamic implementation should run as an isolated Worker or Pages Function and not compromise static blog availability.

Acceptance:

- Page is not a broken placeholder.
- Feature flag or environment detection prevents build failures when AI secrets are absent.

### 8. Analytics, License, Source Links

Requirements:

- Umami script is optional and configured through public env vars.
- Article footer shows license `CC BY-NC-SA 4.0` when enabled.
- Article source/edit link points to GitHub content path when configured.
- All optional external scripts must fail closed: no env var means no script rendered.

Acceptance:

- No secrets in repo.
- Static build works without analytics configuration.

## Suggested Migration Order

1. Frontend redesign shell and weak card system.
2. RSS and search-data static generation.
3. Archive and tags.
4. About and profile page.
5. Friends static page and data model.
6. Books data model and listing.
7. Tools/Lab registries.
8. Ask AI planning page; dynamic AI later.
9. Analytics/license/source links.

## Out of Scope for This Migration Round

- Full AI Q&A with vector search.
- Friend link submission API backed by storage.
- Cloudflare D1/KV/R2/Vectorize setup.
- Reintroducing heavy theme customization from old Fuwari.
- Recreating old dynamic effects such as Three.js wave/chaos backgrounds.

## Acceptance Criteria

- Every migrated page is available under localized `/zh` and `/en` routes.
- SEO-critical pages are statically generated.
- `bun test` and `bun run pages:build` pass after each migration slice.
- No route reintroduces `_worker.js` until OpenNext runtime compatibility is intentionally revisited.
